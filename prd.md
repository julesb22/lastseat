## Executive Summary

LastSeat is a last-minute sports and event ticketing platform that connects event-goers with unsold tickets available a few days before an event takes place. Rather than letting seats go empty, LastSeat surfaces those tickets to nearby users who are ready to attend on short notice.

The core value proposition is simple: **if you're flexible, you get access**. LastSeat removes the friction of last-minute attendance by aggregating unsold inventory and surfacing it based on the user's location. For the MVP, admins manually upload unsold ticket inventory, making the platform immediately usable without requiring third-party integrations.

The primary audience is spontaneous event-goers — people open to attending a sports match, concert, or live event within the next few days if the price and location are right. LastSeat differentiates itself by focusing exclusively on unsold, near-expiry inventory, making it a destination for deals rather than a general ticketing marketplace.

---

## Current Situation

- Event organizers (sports teams, venues, promoters) routinely face unsold inventory in the days leading up to an event.
- Existing platforms like Ticketmaster, StubHub, and SeatGeek are primarily designed for advance purchases — they are not optimized for last-minute, location-aware discovery.
- Consumers who are spontaneous or price-sensitive have no dedicated, friction-free channel to discover and buy last-minute tickets near them.
- Currently, unsold tickets are either discounted at the venue box office, shared through informal channels (social media, WhatsApp groups), or simply go to waste.

---

## Problem Statement

- **Friction in last-minute discovery**: There is no dedicated platform where a user can open an app, see what events are happening nearby in the next few days, and purchase a ticket in minutes.
- **Wasted inventory**: Event organizers lose revenue on unsold tickets that could have been filled at a discount.
- **No location-aware filtering**: Existing tools don't prioritize proximity when surfacing events to buyers.
- **Quantifiable impact**: An empty seat = $0 revenue for the organizer + a missed experience for the potential attendee. LastSeat converts that $0 into a transaction for both sides.
- **Poor UX for spontaneous buyers**: Current platforms are designed for planners, not for last-minute, impulse-driven purchases.

## Solution Overview

### V0 / MVP

- User-facing web application where authenticated users can:
  - View a list of events near their location with unsold tickets available
  - Filter events by type, date range, and distance
  - View event details (venue, date/time, ticket categories, price)
  - Purchase one or more tickets directly in the app
  - Receive a booking confirmation with ticket details
- Admin panel (basic) where an admin can:
  - Manually upload/add new events and unsold ticket inventory
  - Manage existing listings (edit, remove, mark as sold)

---

#### ⚠️ Risk 1: Stripe Payment Webhooks — Medium Risk

**What it is**: When a user pays, Stripe sends a webhook event to confirm the transaction. If this webhook fails or is missed, a user could be charged without receiving their ticket.

**Validation task**:
1. Create a free Stripe account at [stripe.com](https://stripe.com)
2. In test mode, set up a webhook endpoint pointing to your local server
3. Use the [Stripe CLI](https://stripe.com/docs/stripe-cli) to forward webhook events locally: `stripe listen --forward-to localhost:3000/api/webhooks/stripe`
4. Trigger a test payment and confirm the `payment_intent.succeeded` event is received
5. ✅ Success = your endpoint receives the event and logs it correctly

---

#### ⚠️ Risk 2: Geolocation Permission Denied — Low-Medium Risk

**What it is**: If the user denies browser location access, the event listing has no default sort order, which breaks the core UX.

**Mitigation**: Build a manual city/zip code input fallback. If geolocation is denied → show a "Enter your city" prompt before displaying the event list.

**Validation task**: Test in Chrome with geolocation blocked (`Settings → Privacy → Location → Block`) and confirm the fallback input appears.

---

#### ⚠️ Risk 3: Ticketmaster API — Future (V1), Validate Early — Low Risk for MVP

**What it is**: For V1, LastSeat will ingest ticket inventory from the Ticketmaster Discovery API automatically.

**Good news**: The Ticketmaster Discovery API **does exist** and is publicly available at [developer.ticketmaster.com](https://developer.ticketmaster.com). It provides event listings, venue data, availability, and pricing.

**Validation task (do before V1 build)**:
1. Register for a free API key at [developer.ticketmaster.com](https://developer.ticketmaster.com/api-explorer/v2/)
2. Open Postman and run: `GET ``https://app.ticketmaster.com/discovery/v2/events.json?apikey=YOUR_KEY&city=Paris&classificationName=sports`
3. Inspect the response — confirm it returns events with: name, date, venue, price ranges, and availability
4. ✅ Success = the API returns usable event + ticket data

**Note on other APIs**: StubHub (Viagogo) and SeatGeek also have developer APIs. SeatGeek's is the most developer-friendly for integration.

---

#### ⚠️ Risk 4: QR Code Venue Validation — Operational Risk

**What it is**: LastSeat generates QR codes per booking — but venues need a way to scan and validate them at the door. For the MVP prototype, this is an operational concern rather than a technical one.

**MVP mitigation**: QR codes encode the booking ID. Venue staff can validate manually by cross-referencing the booking ID with the admin panel. A proper scanning solution can be built post-MVP.
